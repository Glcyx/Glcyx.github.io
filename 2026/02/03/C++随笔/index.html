1. 构造函数中不应进行ＩＯ操作

  • 构造函数的主要目的是初始化对象的成员变量，而不是执行任何与对象外部交互的操作

    1. 不确定的对象状态：构造函数的执行是在对象创建的过程中进行的，此时对象可能还没有完全初始化。因此，在构造函数中执行 I/O 操作可能导致对象处于不确定的状态，这可能会产生意外的行为或错误。
    2. 可维护性问题：将 I/O 操作放在构造函数中可能会导致代码的可读性和可维护性下降。构造函数应该专注于初始化对象的成员变量，而不应该包含与对象初始化无关的逻辑。
    3. 性能问题：构造函数的主要任务是尽快完成对象的初始化过程。如果在构造函数中执行耗时的 I/O 操作，可能会导致对象创建的时间变长，影响程序的性能。

3. 拷贝构造函数 const问题

  • 在编写拷贝构造函数时，参数列表中是否需要加上 const 取决于你的设计和需求。通常情况下，拷贝构造函数的参数应该是一个常量引用或者普通引用，而不是一个指针。这是因为拷贝构造函数的目的是创建一个新对象并将其初始化为另一个对象的副本，而不是修改原始对象。

    如果你选择使用引用作为参数，建议将其声明为常量引用，这样可以确保不会修改原始对象的状态

  • 如果你不使用 const 修饰符，编译器将会认为你有可能会修改原始对象的状态，这可能会导致不可预料的行为。因此，在编写拷贝构造函数时，最好将参数声明为 const 引用，以确保不会意外修改原始对象。

4 .关于返回类型为引用

  • 当返回类型为引用时,我们可以用引用来接收,也可用普通变量来接收

  • 但倘若用引用对象来接受返回类型为普通变量类型时,就会出错

  • reason:

  • 当我们返回类型为普通类型时，例如int，被调函数结束的时候，会生成一个临时对象，然后return的内容拷贝给这个临时对象。这里为什么要生成临时对象呢？当被调函数执行完后，相应栈空间就会被释放，可能被重新分配，导致里面的内容消失，所以才会另外产生一个临时对象，来存储要返回的内容。当然不一定非要产生一个临时对象，也可以放到寄存器中短暂存储。只要能保证这个内容不会消失即可。然后将这个临时对象返回给主调函数。

  • 但返回类型为引用时，例如int*,不会生成临时对象。会直接返回对象。所以呢，我们绝对不允许返回局部对象的引用。因为等到被调函数执行完毕后，这个局部变量都找不到了，还何谈引不引用？

  • 当返回普通变量时，返回的时临时对象。对临时对象的引用可是非常危险的，所以呢，编译器不允许通过。

  • 出错处相当于: int& c=临时对象；

6.const T &val和const T val

在C++中，const T &val 和 const T val 都是表示参数不应该被修改，但是它们之间有一些细微的差别：

  1. const T &val：这是一个常量引用。当你使用 const T &val 作为参数时，你实际上是在创建一个对传入参数的引用，并且这个引用是常量，意味着你不能通过这个引用来改变原始数据。这种方式的优点是，它允许函数接受非常量对象的引用（通过添加 const），并且避免了对象的复制，这可以提高效率，尤其是当对象是大型数据结构时。

  2. const T val：这是一个常量对象。当你使用 const T val 作为参数时，你实际上是在函数内部创建了一个新的常量对象，它是传入参数的一个副本。这种方式的缺点是，如果传入的参数是一个大型数据结构，那么每次调用函数时都会进行复制，这可能会导致不必要的性能开销。

在大多数情况下，如果你不需要修改传入的数据，并且想要避免不必要的复制，那么 const T &val 是更好的选择。然而，如果你需要确保函数内部的代码不会以任何方式改变传入的数据，并且不介意复制数据，那么 const T val 也是一个可行的选择。

在你提供的代码中，Insert 函数的参数是 const T &val，这是正确的，因为它允许函数接受一个常量引用，避免了不必要的复制，同时确保了传入的值不会被修改。如果你使用 const T val，那么每次调用 Insert 函数时，都会复制传入的值，这可能不是你想要的，尤其是在处理大型数据结构时。

13.using namespace std

14. unique_lock和lock_guard

  • unique_lock 提供了更多的灵活性，允许延迟锁定、尝试锁定、定时锁定以及解锁后重新锁定。

  • lock_guard 则更为简单，一旦构造就会立即锁定互斥锁，并在作用域结束时自动解锁，没有提供延迟锁定或解锁的能力。

  • unique_lock有运算符重载和拷贝构造函数,另一个无,所以unique_lock 可以转移所有权，即可以将一个 unique_lock 对象的所有权转移给另一个 unique_lock 对象。lock_guard 不支持所有权转移。

  • 类比一下scoped_ptr和unique_ptr

16. 栈帧

  • 是调用栈（Call Stack）中的一个单元，它包含了一个函数调用的所有信息。每当一个函数被调用时，一个新的栈帧就会被创建并压入调用栈的顶部。当函数执行完毕后，其对应的栈帧会被弹出栈，释放资源。
  • 包含信息: 返回地址,局部变量,寄存器保存区,栈帧指针(链接栈帧) ….

17.inline关键字

  • 作用:建议编译器将函数的定义（即函数体）直接插入到每个函数调用的地方，而不是进行常规的函数调用。

  • 好处:

    减少函数调用开销：避免了参数传递、栈帧创建和销毁等开销，因为函数体直接插入到调用点，不需要额外的调用过程。

    提高执行效率：对于小型函数，这种内联可以提高程序的执行效率，因为减少了函数调用的开销。

  • 由编译器决定是否内联,也就是说发生在编译阶段

20. 用户栈 (user stack)

  • 用户栈是用户态进程使用的栈，用于存储函数调用的局部变量、返回地址和参数。
  • 每个用户态进程都有自己独立的用户栈。
  • 存储函数调用上下文,支持用户态程序正常执行
  • 如果用户栈溢出,发生 栈溢出 错误

21. 内核栈(kernel stack)

  • 内核栈是内核态代码使用的栈，用于处理系统调用、中断和异常。位于内核空间,与用户空间隔离

  • 当用户程序发起系统调用（如 read）时，CPU 会切换到内核态，并使用内核栈保存上下文。

  • 如果内核栈溢出，会导致 系统崩溃（Kernel Panic）。

26. 常量表达式,constexpr 和 const

  1. constexpr函数

  • constexpr 函数是指能用于常量表达式的函数。

  • constexpr 函数既能用于要求常量表达式/编译期常量的语境，也可以作为普通函数使用。

    注意：constexpr 函数不一定返回常量表达式！

  • 只有 constexpr 的所有实参都是常量表达式/编译期常量时，constexpr 函数的结果才是常量表达式/编译期常量。只要有一个参数在编译期未知，那就和普通函数一样，在运行时计算。

  1. 好处

  • 编译器可以保证 constexpr 对象是常量表达式（能够在编译期取得结果），而 const 对象不能保证。如果一个 const 变量能够在编译期求值，将其改为 constexpr 能够让代码更清晰易读
  • constexpr 函数可以把运行期计算迁移至编译期，使得程序运行更快（但会增加编译时间）

对于常量表达式（编译期值已知），编译器可以进行更多优化，比如放到只读内存中。但这并不是 constexpr 特有的，有的 const 变量也是常量表达式

29.指针的初始化

指针的声明

  • 在C++中，声明一个指针变量（如ListNode* cur;）时，如果没有显式地初始化它，那么它的值是未定义的。这意味着cur可能指向一个随机的内存地址。
  • 当你尝试访问cur->next时，实际上是在访问一个随机地址的next成员，这会导致未定义行为（Undefined Behavior, UB），可能会导致程序崩溃、错误的结果，甚至看似正常运行但隐藏着潜在问题。

访问随机地址

当你尝试访问cur->next时，实际上是在尝试解引用cur，即访问cur所指向的内存地址中的内容。然而，由于cur未初始化，它的值是随机的，你无法预测它指向哪里。这可能导致以下几种情况：

  • 访问合法内存：如果cur恰好指向一个合法的内存区域，程序可能看起来正常运行，但这只是巧合。
  • 访问非法内存：如果cur指向一个非法的内存区域（如已释放的内存或未分配的内存），程序可能会崩溃（如段错误，Segmentation Fault）。
  • 访问受保护的内存：如果cur指向一个受保护的内存区域（如操作系统内核空间），程序可能会被操作系统终止。

栈是以底层容器完成其所有的工作，对外提供统一的接口，底层容器是可插拔的（也就是说我们可以控制使用哪种容器来实现栈的功能）。

栈的底层实现可以是list,vector,deque,栈提供push 和 pop 等等接口，所有元素必须符合先进后出规则，所以栈不提供走访功能，也不提供迭代器(iterator)。 不像是set 或者map 提供迭代器iterator来遍历所有元素

队列中先进先出的数据结构，同样不允许有遍历行为，不提供迭代器, SGI STL中队列一样是以deque为缺省情况下的底部结构

31. 容器适配器(Container Adapters)

在C++标准库中确实是由底层容器实现的。容器适配器是一种特殊的容器，它们基于现有的容器类型（如 std::vector、std::deque,std::list,），并通过限制或扩展这些容器的功能来提供特定的接口和行为。

容器适配器不支持直接迭代（即不能使用基于范围的 for 循环或普通迭代器）

C++标准库中常见的容器适配器包括：

  1. std::stack(栈)：默认使用 std::deque 作为底层容器。
  2. std::queue(队列)：默认使用 std::deque 作为底层容器。
  3. std::priority_queue(优先级队列)：默认使用 std::vector 作为底层容器。

  1. 隐藏底层容器的细节：容器适配器只暴露了与特定功能相关的接口，隐藏了底层容器的直接操作。
  2. 提供特定的接口：例如，std::stack 只提供 push()、pop()、top() 等栈操作接口；std::queue 提供 push()、pop()、front()、back() 等队列操作接口。
  3. 依赖底层容器的实现：容器适配器的行为和性能特性依赖于底层容器的选择。

32.随机访问

随机访问 指的是能够直接访问数据结构中的任意元素，而无需按顺序遍历前面的元素。支持随机访问的容器（如数组、std::vector、std::deque,std::string）允许通过**索引（下标）*或*迭代器算术运算（如 +、-）快速访问任意位置的元素。

  • 时间复杂度 O(1)：可以直接通过索引或迭代器计算跳转到目标位置，无需逐个遍历。
  • 支持迭代器算术运算：如 iter + n、iter - n、iter += offset 等。
  • 支持下标运算符 []：如 arr[3] 或 vec[5]。

重载(overload)与重写(override)

一、重载（overload）
指函数名相同，但是它的参数表列个数或顺序，类型不同。但是不能靠返回类型来判断。
（1）相同的范围（在同一个作用域中） ；
（2）函数名字相同；
（3）参数不同；
（4）virtual 关键字可有可无。
（5）返回值可以不同；

二、重写（也称为覆盖 override）
是指派生类重新定义基类的虚函数，特征是：
（1）不在同一个作用域（分别位于派生类与基类） ；
（2）函数名字相同；
（3）参数相同；
（4）基类函数必须有 virtual 关键字，不能有 static 。
（5）返回值相同（或是协变），否则报错；
(6）重写函数的访问修饰符可以不同。尽管 virtual 是 private 的，派生类中重写改写为 public,protected 也是可以的
