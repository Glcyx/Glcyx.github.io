发表于 2026-02-27 更新于 2026-04-20 分类于 MLIR

CSE.cpp

这个pass也非常常见,平时经常就是canonicalize后面跟一个cse

整体结构

目的:

CSE也就是Common Subexpression Elimination

核心目标就是消除重复计算

如果两个op

  • 操作类型相同
  • 操作数相同
  • 属性相同
  • 没有会影响结果的副作用

那么后一个op就没必要再算一次,直接复用前一个结果就行

命令行参数

执行命令就是

一个最简单的理解

例如

如果这两个op都没有副作用,那么第二个就可以删掉,把%1的使用点都替换成%0

这个思想本身不复杂,但真正做的时候要考虑:

  • 支配关系
  • region / block 层级
  • 带副作用的op不能乱合并

核心逻辑

不是简单全局哈希一下就结束

第一眼可能会觉得

那不就是给op做个hash,一样的就替换

但实际上不行

因为前面的op必须能支配后面的op

也就是说 你要替换一个结果,前提是原来那个结果在控制流上对当前点可见

所以这个pass要结合遍历顺序和作用域结构来做

遍历方式

这个文件里会递归处理 operation / region / block

因为MLIR IR本身就是嵌套结构,不是简单一层线性指令

所以CSE不能只看一个block里的相邻语句,而是要在合法支配范围内维护“当前已经见过且可复用的表达式”

hasNoSideEffect 这种判断非常关键

重复表达式消除有个前提

这个op要足够“纯”

比如:

  • 常量
  • 加法
  • 一些纯计算

这些是可以合并的

但如果一个op会:

  • 写内存
  • 读取某些会变化的状态
  • 产生外部可观察副作用

那就不能简单认为“长得一样所以结果一样”

这也是CSE和普通文本去重完全不是一回事的原因

值得注意的地方

block argument 不能乱动

CSE主要处理的是重复op

而不是把所有值都拿来乱合并

像BlockArgument这种值来源就不是某个具体op,所以处理逻辑本来就不同

region嵌套时作用域会变化

MLIR里一个operation内部还能套region

所以进入一个新的region时,你要考虑:

  • 外层有哪些值是可见的
  • 当前层新增了哪些可候选的表达式
  • 离开作用域后哪些东西不能继续拿来复用

这个文件一个比较重要的点就是把这种嵌套结构里的可用表达式管理起来

死代码删除会顺手发生

当后一个重复op被替换掉以后,它自己就没用了

这时候就可以被删掉

所以CSE看起来只是“合并重复表达式”,实际上最后效果经常会让IR明显短一截

为什么经常接在canonicalize后面

因为canonicalize经常会把IR整理成更规范的形式

比如原本写法不同但语义等价的东西,先被整理成统一样子

这时候再跑CSE,就更容易识别出“你们两个其实一样”

所以顺序上:

很常见

如果中间又生成了新的可简化形式,后面甚至再补一个canonicalize

对自己写优化有什么帮助

我觉得看这个文件主要是提醒自己:

第一,不是所有重复长相的op都能合并

要先看支配关系和副作用

第二,优化pass之间是会互相喂机会的

一个pass并不需要把所有事情做完

它只要把IR往更规范、更简洁的方向推一步,后面的pass就可能继续接力

第三,MLIR里很多“常规优化”其实已经有现成pass

所以自己写pipeline的时候,不用每次都从零开始造轮子

像

  • canonicalize
  • cse

这种基础pass先接上,很多中间垃圾会自己少很多

一个整体印象

CSE.cpp读下来没有那种特别花的技巧,但很典型

它体现的是编译器里一个很基础但是很重要的思路:

在保证语义不变的前提下,不要重复做同样的计算

而MLIR把这件事放在嵌套IR结构里来做,所以比单block文本理解要更有意思一点
