CUDA Kernel 有效带宽 (Effective Bandwidth) 的计算

有效带宽的计算公式通常是：
$$
\text{Effective Bandwidth} = \frac{\text{Data Transferred}}{\text{Time} \times 10^9} \quad \text{(GB/s)}
$$
其中：

  • Data Transferred (数据传输量)：指 Kernel 执行期间从全局内存（Global Memory）读取和写入的总字节数（Read Bytes + Written Bytes）。
  • Time (时间)：指 Kernel 的执行时间（以秒为单位）。
  • 10^9：用于将结果从 B/s 转换为 GB/s。

内存事务: 从核函数发起内存请求到硬件响应返回数据的过程

当一个内存事务的首个访问地址是缓存粒度（L1:32B(byte)或L2:128B）的偶数倍的时候：比如二级缓存32字节的偶数倍64，128字节的偶数倍256的时候，这个时候被称为对齐内存访问，非对齐访问就是除上述的其他情况，非对齐的内存访问会造成带宽浪费。

比如说落在32,总比落在33好,万一你要访问32B的数据,那么落在32就只用访问一次,落在33就要访问两次

“落在边界”只能保证不会因为对齐问题产生额外的事务，但不能保证整个访问一定只需要一次事务。

原因是：
是否只需要一次事务，还取决于访问范围是否跨越事务大小。

合并访问(coalescing): 一个线程束的所有线程内存访问都在同一个内存块

内存事务的优化关键：用最少的事务次数满足最多的内存请求。事务数量和吞吐量的需求随设备的计算能力变化。

缓存

  1. 一级缓存
  2. 二级缓存
  3. 只读常量缓存
  4. 只读纹理缓存

每个SM都有一个一级缓存,只读常量缓存,只读纹理缓存，所有SM公用一个二级缓存。

作用: 一二级缓存都是被用来存储本地内存和全局内存中的数据，也包括寄存器溢出的部分。

数据访问过程:

当线程需要访问某个地址时，首先检查寄存器，若不在寄存器，则访问 L1 缓存；地址通过索引计算定位到某个 cache set 并与该 set 的所有 tag 并行比较，若命中则直接将数据返回寄存器。若 L1 未命中，则以相同方式查询 L2 缓存，若命中，则数据会送回寄存器，同时可能回填到 L1。若 L2 仍未命中，则从显存 DRAM 中按缓存行(颗粒度)（有 L1 时为 128B，无 L1 时为 32B）取回整块数据，并回填到 L2（以及可能回填到 L1,当L1启动时），随后将其中真正需要的部分送入寄存器供线程使用；剩余部分留在缓存供后续访问命中。

内存加载可以分为两类：

  • 缓存加载
  • 没有缓存的加载

内存访问特点:

  • 是否使用缓存(一级缓存是否介入,可手动控制)
  • 对齐与非对齐
  • 合并与非合并

内存块(memory block)是你在 DRAM 中的数据，而缓存行(cache line)是硬件在把 DRAM 数据搬到缓存时切成的固定大小的“运输单元”。

  • 对齐访问：访问落进单一缓存行
  • 非对齐访问：访问跨越多个缓存行 → 需要更多事务 → 带宽浪费

缓存加载

  1. 对齐合并访问,内存利用率100%
  2. 对齐,不连续,warp内线程访问位置在一个粒度内,位置交叉,利用率100%
  3. 连续非对齐, 当访问32个4B数据,那么数据会横跨两个块,利用率(32 * 4)/(128 * 2) = 50%
  4. warp线程访问同一位置,如果是4B大小,利用率4/128=3.125%

没有缓存的加载

没有通过L1, L2是不得不通过的

粒度变为32B,好处是提高了内存利用率

  1. 对齐合并访问
  2. 对齐不连续
  3. 连续非对齐, 如果同样是32*4B的数据,即使不对其,最多访问5个块(因为连续),利用率32 * 4 / 32 * 5 = 80%
  4. 所有线程访问一个4B数据,利用率 4/32 = 12.5%

只读缓存(read only):
颗粒度 32B,对于分散读取,细粒度优于L1(128B)

全局内存写入:
NVIDIA GPU 的全局内存写入（global memory store）默认不经过 L1 Cache，只经过 L2 Cache。

原因: 全局写入一般不需要读回数据,绕过L1,减少对其污染,为读取提高命中率

从 Maxwell开始,L1和shared memory 合并成了共享结构,但是写入(store)仍然不经过L1部分,只是用shared memory路径,他们只共享物理块,路径仍分离

对于AoS(数组结构体)和SoA(结构体数组)

cuda倾向于SoA,因为这种内存访问可以有效地合并

矩阵转置

当然,所有的数据在内存里都是一维排布的,不管你是数组,结构体,多维数组..

不难看出:

  • 读: 请求内存连续,线程束可以连续访问(相同颜色可以看成合并读取)
  • 写:写到转置矩阵的列中,访问是交叉的

实际上 先列读取再行写入效率会高一些 因为列读一级缓存一次读取128字节的数据，其中只有一个单位有用，但是剩下的并不会被马上覆盖，粒度是128字节，但是一级缓存的大小有几k或是更大，这些数据很有可能不会被替换，所以，我们按列读取数据，虽然第一行只用了一个，但是下一列的时候，理想情况是所有需要读取的元素都在一级缓存中，这时候，数据直接从缓存里面读取,速度变快 也就是缓存命中

对角转置：读取行与读取列

说是有性能优化

共享内存

是一种可编程的缓存,作用:

  1. 块内线程通信的通道
  2. 用于全局内存数据的可编程管理的缓存
  3. 告诉暂存存储器，用于转换数据来优化全局内存访问模式

在其所属线程块被执行时建立, 在线程块执行完毕后释放, 线程块与其共享内存拥有相同的生命周期.

SM占有固定大小共享内存,分配block一定大小shared memory, 当block占有的shared memory越多,SM可支配的block就越少(因为SM拥有的SMEM大小是固定的),那么warp数量也就越少,导致延迟隐藏能力下降,减少吞吐量

现在所有NVIDIA GPU中的共享内存(SMEM)都被划分为32个存储体(banks),与warp中32个线程对应

每个存储体可同时为一个线程提供访问,默认大小4B(byte)

存储体访问模式:

  1. 并行访问: 多线程访问多存储体
  2. 串行访问: 多线程访问同一存储体 发生内部冲突(bank conflict)
  3. 广播访问: 多个线程访问同一个地址,不会发生冲突,而是硬件统一执行一次读取,将结果广播给其他访问该地址的线程的寄存器

在访问shared memory时，因多个线程读写同一个Bank中的不同数据地址时，导致shared memory 并发读写 退化 成顺序读写的现象叫做Bank Conflict；

需要注意的是，不同 warp 中的线程之间不存在 bank 冲突。

内存填充（padding）

多开了一列,但是bank数量必须是32,所以把多的那一列往后推,使得bank原先每一列的数据错位,避免了冲突,也就是说 :

你声明一个二维存储体，编译器会把声明的二维转换成一维线性的，然后再重新整理成二维按照32个存储体，4-Byte/8-Byte宽的内存分布，然后再进行运算的

优点在于：代码改动少，易于理解；而缺点有：

  1. 增加了每个 block 用到的 shared memory 空间，可能导致 SM 内最大可并发的 block 数目减少，降低 occupancy。
  2. padding 后会出现地址不对齐的情况，如果采用如 float4 这样的向量化访问，会导致执行出错。

弱排序内存模型

GPU 中的线程对内存的读写不保证按照源代码顺序执行；只要两个内存访问指令之间没有依赖，硬件可能乱序执行它们，并且不同线程看到的写入顺序也可能不一致。因此如果多个线程需要对共享数据进行有序、可控的读写，就必须使用同步手段，否则会出现完全不可预测的竞争行为。

内存栅栏

内存栅栏能保证栅栏前的内核内存写操作对栅栏后的其他线程都是可见的，有以下三种栅栏：块，网格，系统。

可见: 防止写操作乱序,确保写入完成,保证其他线程读取的是最新值

线程块内：

保证同一块中的其他线程对于栅栏前的内存写操作可见

网格级内存栅栏:

挂起调用线程，直到全局内存中所有写操作对相同的网格内的所有线程可见

系统级栅栏，跨系统，包括主机和设备:

挂起调用线程，以保证该线程对全局内存，锁页主机内存和其他设备内存中的所有写操作对全部设备中的线程和主机线程可见。

Volatile修饰符

volatile声明一个变量，防止编译器优化，防止这个变量存入缓存，如果恰好此时被其他线程改写，那就会造成内存缓存不一致的错误，所以volatile声明的变量始终在全局内存中。

流和事件

cuda流: 一系列的cuda异步操作 例如 host分配内存/传输数据给device cudaMalloc / cudaMemcpy, kernel启动 传输数据回host Memcpy

流可以对这些操作进行封装 , 保持执行顺序 允许操作在流中排队

流中的操作对于host来说总是异步的, 因为cuda决定何时进行这些操作,我们要控制这些操作在其结果出来前调用需要其结果的操作, 避免死锁

流内操作有严格顺序 但流间没有限制,多个流同时启动多个核,形成了网络grid级别的并行

cuda的API也分同步异步

  • 同步行为函数会阻塞host线程 直至完成
  • 异步行为函数会再调用后立刻把指挥权返还host

所有cuda操作都在流中进行,流分为:

  • 隐式声明的流, 称为空流 (默认) 没有名字 无法管理
  • 显式声明的流, 称为非空流

基于流的异步内核启动和数据传输支持以下粗粒度并发 (重叠就是同时进行)

类型 含义
重叠主机和设备计算 CPU 与 GPU kernel 同时跑
重叠主机计算和数据传输 CPU 与 H↔D memcpy 同时跑
重叠数据传输和设备计算 GPU 的 kernel 与 GPU 的 DMA 拷贝同时进行
并发设备计算 多个 GPU 同时跑自己的 kernel

优化一:减少warp divergence

Warp Divergence 是指在一个 warp 中，不同线程执行不同路径（即不同的条件分支）

优化二: 解决bank conflicts

这里提醒一下: 因为发生冲突条件是一个warp里的不同线程访问同一个bank的不同位置
所以说在存数据的时候,一个warp刚好存一行,一共八个warp,存八行,此时不会发生冲突

优化后,0号线程读取0,128,1号线程读取1,129,情况是同一个线程读取同一个bank里的数据,这种情况不是冲突,按照知乎说法是串行执行

优化二中,每个block中,128-255线程除了赋值,一次加法也不做,为了合理利用,我们在复制的时候就进行一次加法运算,当然,前提是扩大单个block负责的数据范围,同时减少block数量
