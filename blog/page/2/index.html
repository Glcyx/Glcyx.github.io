但实际上不行

这个op要足够“纯”

  • 常量
  • 加法
  • 一些纯计算

这些是可以合并的

  • 写内存
  • 读取某些会变化的状态
  • 产生外部可观察副作用

这也是CSE和普通文本去重完全不是一回事的原因

  • 离开作用域后哪些东西不能继续拿来复用

整体结构

目的:

这个文件实现的是MLIR里“贪心pattern重写”的驱动逻辑

也就是:

  • 你把一堆pattern塞进去
  • 它负责不断找能匹配的op
  • 一旦匹配成功就重写
  • 重写后继续看有没有新的机会
  • 直到整个IR收敛

所以它本身不是某个具体优化,而是一个基础设施

很多pass其实都只是:

  1. 准备pattern
  2. 调这个driver

为什么叫greedy

因为它的策略不是全局最优搜索,而是

看到当前能改的,就先改

改完以后再看周围还能不能继续改

这种方式实现简单,而且对大多数局部重写场景很有效

可以怎么理解这个driver

我觉得可以把它理解成一个“工作队列 + 不断回头看”的过程

第一步: 初始化工作列表

先把当前作用范围里的op收集起来

这些op就是潜在候选

第二步: 依次尝试匹配pattern

对每个候选op

  • 看有没有pattern能匹配
  • 如果能匹配,就执行matchAndRewrite

第三步: IR变了以后,附近op重新入队

这个点很关键

因为一次重写之后

  • 当前op可能没了
  • 新op可能产生了
  • 使用者可能变了
  • 定义者附近也可能出现新机会

所以driver不是“一次遍历所有op”就结束

而是会把受影响区域重新放回工作列表

第四步: 直到没有pattern再能匹配

这时就收敛了

也就是所谓的fixpoint

1. 构造函数中不应进行ＩＯ操作

  • 构造函数的主要目的是初始化对象的成员变量，而不是执行任何与对象外部交互的操作

    1. 不确定的对象状态：构造函数的执行是在对象创建的过程中进行的，此时对象可能还没有完全初始化。因此，在构造函数中执行 I/O 操作可能导致对象处于不确定的状态，这可能会产生意外的行为或错误。
    2. 可维护性问题：将 I/O 操作放在构造函数中可能会导致代码的可读性和可维护性下降。构造函数应该专注于初始化对象的成员变量，而不应该包含与对象初始化无关的逻辑。
    3. 性能问题：构造函数的主要任务是尽快完成对象的初始化过程。如果在构造函数中执行耗时的 I/O 操作，可能会导致对象创建的时间变长，影响程序的性能。

4 .关于返回类型为引用

  • 当返回类型为引用时,我们可以用引用来接收,也可用普通变量来接收

  • 但倘若用引用对象来接受返回类型为普通变量类型时,就会出错

  • reason:

  • 当我们返回类型为普通类型时，例如int，被调函数结束的时候，会生成一个临时对象，然后return的内容拷贝给这个临时对象。这里为什么要生成临时对象呢？当被调函数执行完后，相应栈空间就会被释放，可能被重新分配，导致里面的内容消失，所以才会另外产生一个临时对象，来存储要返回的内容。当然不一定非要产生一个临时对象，也可以放到寄存器中短暂存储。只要能保证这个内容不会消失即可。然后将这个临时对象返回给主调函数。

  • 但返回类型为引用时，例如int*,不会生成临时对象。会直接返回对象。所以呢，我们绝对不允许返回局部对象的引用。因为等到被调函数执行完毕后，这个局部变量都找不到了，还何谈引不引用？

  • 当返回普通变量时，返回的时临时对象。对临时对象的引用可是非常危险的，所以呢，编译器不允许通过。

  • 出错处相当于: int& c=临时对象；

6.const T &val和const T val

在C++中，const T &val 和 const T val 都是表示参数不应该被修改，但是它们之间有一些细微的差别：

  1. const T &val：这是一个常量引用。当你使用 const T &val 作为参数时，你实际上是在创建一个对传入参数的引用，并且这个引用是常量，意味着你不能通过这个引用来改变原始数据。这种方式的优点是，它允许函数接受非常量对象的引用（通过添加 const），并且避免了对象的复制，这可以提高效率，尤其是当对象是大型数据结构时。

  2. const T val：这是一个常量对象。当你使用 const T val 作为参数时，你实际上是在函数内部创建了一个新的常量对象，它是传入参数的一个副本。这种方式的缺点是，如果传入的参数是一个大型数据结构，那么每次调用函数时都会进行复制，这可能会导致不必要的性能开销。

在大多数情况下，如果你不需要修改传入的数据，并且想要避免不必要的复制，那么 const T &val 是更好的选择。然而，如果你需要确保函数内部的代码不会以任何方式改变传入的数据，并且不介意复制数据，那么 const T val 也是一个可行的选择。

在你提供的代码中，Insert 函数的参数是 const T &val，这是正确的，因为它允许函数接受一个常量引用，避免了不必要的复制，同时确保了传入的值不会被修改。如果你使用 const T val，那么每次调用 Insert 函数时，都会复制传入的值，这可能不是你想要的，尤其是在处理大型数据结构时。

16. 栈帧

  • 是调用栈（Call Stack）中的一个单元，它包含了一个函数调用的所有信息。每当一个函数被调用时，一个新的栈帧就会被创建并压入调用栈的顶部。当函数执行完毕后，其对应的栈帧会被弹出栈，释放资源。
  • 包含信息: 返回地址,局部变量,寄存器保存区,栈帧指针(链接栈帧) ….

17.inline关键字

  • 作用:建议编译器将函数的定义（即函数体）直接插入到每个函数调用的地方，而不是进行常规的函数调用。

  • 好处:

    减少函数调用开销：避免了参数传递、栈帧创建和销毁等开销，因为函数体直接插入到调用点，不需要额外的调用过程。

    提高执行效率：对于小型函数，这种内联可以提高程序的执行效率，因为减少了函数调用的开销。

  • 由编译器决定是否内联,也就是说发生在编译阶段

20. 用户栈 (user stack)

  • 用户栈是用户态进程使用的栈，用于存储函数调用的局部变量、返回地址和参数。
  • 每个用户态进程都有自己独立的用户栈。
  • 存储函数调用上下文,支持用户态程序正常执行
  • 如果用户栈溢出,发生 栈溢出 错误

21. 内核栈(kernel stack)

  • 内核栈是内核态代码使用的栈，用于处理系统调用、中断和异常。位于内核空间,与用户空间隔离

  • 当用户程序发起系统调用（如 read）时，CPU 会切换到内核态，并使用内核栈保存上下文。

29.指针的初始化

指针的声明

  • 在C++中，声明一个指针变量（如ListNode* cur;）时，如果没有显式地初始化它，那么它的值是未定义的。这意味着cur可能指向一个随机的内存地址。
  • 当你尝试访问cur->next时，实际上是在访问一个随机地址的next成员，这会导致未定义行为（Undefined Behavior, UB），可能会导致程序崩溃、错误的结果，甚至看似正常运行但隐藏着潜在问题。

访问随机地址

当你尝试访问cur->next时，实际上是在尝试解引用cur，即访问cur所指向的内存地址中的内容。然而，由于cur未初始化，它的值是随机的，你无法预测它指向哪里。这可能导致以下几种情况：

  • 访问合法内存：如果cur恰好指向一个合法的内存区域，程序可能看起来正常运行，但这只是巧合。
  • 访问非法内存：如果cur指向一个非法的内存区域（如已释放的内存或未分配的内存），程序可能会崩溃（如段错误，Segmentation Fault）。
  • 访问受保护的内存：如果cur指向一个受保护的内存区域（如操作系统内核空间），程序可能会被操作系统终止。

\text{Effective Bandwidth} = \frac{\text{Data Transferred}}{\text{Time} \times 10^9} \quad \text{(GB/s)}
$$
其中：

  • Data Transferred (数据传输量)：指 Kernel 执行期间从全局内存（Global Memory）读取和写入的总字节数（Read Bytes + Written Bytes）。
  • Time (时间)：指 Kernel 的执行时间（以秒为单位）。
  • 10^9：用于将结果从 B/s 转换为 GB/s。

数据访问过程:

当线程需要访问某个地址时，首先检查寄存器，若不在寄存器，则访问 L1 缓存；地址通过索引计算定位到某个 cache set 并与该 set 的所有 tag 并行比较，若命中则直接将数据返回寄存器。若 L1 未命中，则以相同方式查询 L2 缓存，若命中，则数据会送回寄存器，同时可能回填到 L1。若 L2 仍未命中，则从显存 DRAM 中按缓存行(颗粒度)（有 L1 时为 128B，无 L1 时为 32B）取回整块数据，并回填到 L2（以及可能回填到 L1,当L1启动时），随后将其中真正需要的部分送入寄存器供线程使用；剩余部分留在缓存供后续访问命中。

共享内存

是一种可编程的缓存,作用:

  1. 块内线程通信的通道
  2. 用于全局内存数据的可编程管理的缓存
  3. 告诉暂存存储器，用于转换数据来优化全局内存访问模式

在其所属线程块被执行时建立, 在线程块执行完毕后释放, 线程块与其共享内存拥有相同的生命周期.

SM占有固定大小共享内存,分配block一定大小shared memory, 当block占有的shared memory越多,SM可支配的block就越少(因为SM拥有的SMEM大小是固定的),那么warp数量也就越少,导致延迟隐藏能力下降,减少吞吐量

现在所有NVIDIA GPU中的共享内存(SMEM)都被划分为32个存储体(banks),与warp中32个线程对应

每个存储体可同时为一个线程提供访问,默认大小4B(byte)

存储体访问模式:

  1. 并行访问: 多线程访问多存储体
  2. 串行访问: 多线程访问同一存储体 发生内部冲突(bank conflict)
  3. 广播访问: 多个线程访问同一个地址,不会发生冲突,而是硬件统一执行一次读取,将结果广播给其他访问该地址的线程的寄存器

在访问shared memory时，因多个线程读写同一个Bank中的不同数据地址时，导致shared memory 并发读写 退化 成顺序读写的现象叫做Bank Conflict；

需要注意的是，不同 warp 中的线程之间不存在 bank 冲突。

内存填充（padding）

多开了一列,但是bank数量必须是32,所以把多的那一列往后推,使得bank原先每一列的数据错位,避免了冲突,也就是说 :

你声明一个二维存储体，编译器会把声明的二维转换成一维线性的，然后再重新整理成二维按照32个存储体，4-Byte/8-Byte宽的内存分布，然后再进行运算的

优点在于：代码改动少，易于理解；而缺点有：

  1. 增加了每个 block 用到的 shared memory 空间，可能导致 SM 内最大可并发的 block 数目减少，降低 occupancy。
  2. padding 后会出现地址不对齐的情况，如果采用如 float4 这样的向量化访问，会导致执行出错。

弱排序内存模型

GPU 中的线程对内存的读写不保证按照源代码顺序执行；只要两个内存访问指令之间没有依赖，硬件可能乱序执行它们，并且不同线程看到的写入顺序也可能不一致。因此如果多个线程需要对共享数据进行有序、可控的读写，就必须使用同步手段，否则会出现完全不可预测的竞争行为。

Caller-Saved Registers（调用者保存寄存器）：调用者负责保存这些寄存器的值，因为被调用者可能会修改它们。如果调用者在调用函数之前希望保留这些寄存器的值，它们就需要在调用前显式地保存。

中间知识,关于机器数,真值,原码,补码,反码

  • 一个数在计算机中的二进制表示形式，叫做这个数的机器数。机器数是带符号的，在计算机用机器数的最高位存放符号，正数为0，负数为1
  • 将带符号位的机器数对应的真正数值称为机器数的真值
  • 比如000000001,真值为+1,10000001,真值为-1.其形式值为129
  • 原码与机器数表示方式一样
  • 反码: 正数的反码是其本身,负数的反码是在原码的基础上符号位不变,其余各个位置取反. 例如: [+1]=[00000001]原= [00000001]反,[-1]=[10000001]原= [11111110]反
  • 补码: 正数补码就是它本身,负数的补码,符号位不变,其余各位取反,最后整体+1,也就是反码+1

  • 可以理解为是否为零,是就是1 不是就是0

11.2 PF标志

  • 第二位是PF,奇偶标志位
  • 记录相关指令执行后,其结果所有的bit位中1的个数是否为偶数,1的个数为偶数,pf=1,如果是奇数,则为0

11.3 SF标志

  • 第七位是SF，符号标志位
  • 记录相关指令执行后,其结果是否为负,结果为负,sf=1,非负(>=0),sf=0
  • 我们将数据当作有符号数来运算时,SF有意义,而将数据当作无符号数运算时,SF就没有意义,运算会影响sf值,关键在于我们如何看待这个值

11.4 CF标志

  • 第零位是CF,进位标志位

  • inc和loop指令不影响CF位

11.5 OF标志

  • 在进行有符号数运算时,可能发生溢出造成结果错误,CPU要对指令执行后是否产生溢出记录
  • 第十一位是OF,溢出标志位
  • 记录有符号数运算的结果是否产生溢出,发生溢出,OF＝１，没有则为０
  • CF和OF表示的进位和溢出分别对无符号数和有符号数运算而言的,它们之间没有关系

第12章 内中断

12.1 内中断的产生

  • 8086CPU有四种情况产生相应的中断信息
    1. 除法错误,例如div指令产生的除法溢出
      1. 单步执行
      2. 执行into指令
      3. 执行int指令
  • 中断类型码标识中断信息来源,是一个字节型数据,可表示256种中断信息来源(中断源)
  • 上述四种中断源的中断类型码分别是
    1. 除法错误: 0
      1. 单步执行: 1
      2. 执行into指令: 4
      3. 执行int指令,int n中的n为字节型立即数,提供中断类型码

  • 中断向量表:中断处理程序的入口地址的列表

12.11 单步中断

  • CPU执行完一条指令后,检测到TF位为1,则产生单步中断,引发中断过程,单步中断类型码是1

  • 中断过程:

    1. 取得中断类型码 1
      1. 标志寄存器入栈,TF,IF设置为０
      2. CS，IP入栈
      3. （IP)=(1*4),(CS）＝（１＊４＋２）

  • TF =0 的目的是,避免CPU在执行中断处理程序的时候发生单步中断,则程序会一直处在执行单步处理程序第一条指令的循环中

  • debug使用t命令执行指令时,将TF设为1,使得CPU工作处于单步中断方式下

12.12 响应中断的特殊情况

  • 存在发生中断,但是不响应的情况
  • 举个例子: 在执行向ss寄存器传送完数据后,即使发生中断,也不响应,主要原因是此时ss:sp指向栈顶,对其设置应该连续完成,若在执行完设置ss指令后CPU响应中断，引发中断过程，则要向栈中压入标志寄存器,CS,IP的值，那么此时ss改变而sp未改变,ss:sp指向的就不是正确的值,引发错误,因此CPU在执行ss改变指令后不中断,提供了一个连续设置ss,sp的时机,我们理应把握住这个时机

第14章 端口

  • 和CPU通过总线相连的芯片除存储器外还有
    1. 各种接口卡(网卡,显卡等)的接口芯片,他们控制接口卡进行工作
      1. 主板上的接口芯片,CPU通过其对部分外设访问
      2. 其他芯片,用来存储相关系统信息,或进行相关输入输出处理
  • 芯片中会有一组CPU读写的寄存器，物理上存在于芯片中，但是与CPU总线相连,CPU对其读写时,通过控制线向它们所在芯片发出端口读写指令
  • 对于CPU而言,将这些寄存器都当作端口,统一编址,建立端口地址空间,每个端口都有对应地址空间
  • CPU可以直接读写以下３个地方的数据
    1. CPU内部寄存器
      1. 内存单元
      2. 端口

14.1 端口的读写

  • PC系统最多定位64KB个不同端口,端口地址范围:0~65535
  • 端口的读写指令只有in 和out 分别从端口读取和写入数据
  • 不能对端口用mov,push,pop等内存读写指令
  • 在使用in out 指令时,只能用 ax al来存放从端口中读入的数据,或要发送至端口的数据,访问8位端口用al,访问16位用ax
  • in al,20h从20h端口读入一个字节
  • out 3f8h,al 往3f8h端口写入一个字节

14,2 CMOS RAM芯片

    1. 包含一个实时钟和一个有128个存储单元的RAM存储器
      1. 芯片靠电池供电，关机后可正常工作
      2. RAM,0~0dh保存时间信息,其余大部分单元保存系统配置信息,供系统启动时BIOS读取
      3. 芯片内部两个端口,地址为70h,71h,CPU通过其访问CMOS
      4. 70h为地址端口,存放要访问的CMOS RAM单元的地址,也就是访问自身寄存器的单元地址. 71h为数据端口,存放RAM读取或要写入的数据

14.3 shl和shr指令

  • shl:
    1. 将一个寄存或内存单元中的数据向左移位
      1. 将最后移出的一位写入CF中
      2. 最低位用0补充
  • 若移动位数大于1,则将移动位数放入cl中
  • 可知X逻辑左移，相当于X＝X(10) * 2
  • **shr: **
    1. 将一个寄存或内存单元中的数据向右移位
      1. 将最后移出的一位写入CF中
      2. 最高位用0补充
  • 若移动位数大于1,则将移动位数放入cl中
  • X右移一位,相当于X＝X(10) /2

14.4 CMOS RAM中存储的时间信息

  • 秒:0 分:2 时:4 日:7 月:8 年:9

      • sti 设置IF=1

        • 必须响应

        • 8086CPU不可屏蔽中断的中断类型码固定为2

        • 中断过程
